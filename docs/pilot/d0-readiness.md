# 파일럿 D0 준비 게이트 — 계측·배포 검증 + 파운더 체크리스트

파일럿(n=2 단일 블라인드, ADR-0004)이 두 제품 질문에 깨끗한 신호를 내려면 D0(**2026-07-22**) 전에 계측·배포가 무결해야 한다. 계측 갭 하나가 n=2 실험 전체를 무효로 만든다. 파일럿 배포 타깃은 웹에서 **네이티브 Android APK로 역전**됐다(지도 [#129](https://github.com/woosung-dev/cookmark/issues/129) · 스펙 [#140](https://github.com/woosung-dev/cookmark/issues/140); ADR-0005 배포 타깃 역전 기록은 #145). 이 문서는 네이티브 기준 계측·배포 준비 상태와 파운더가 D0에 밟을 절차를 한 곳에 모은다.

## 검증 결과 (네이티브 기준 · 전부 그린)

### 계측 — P2 킬 기준의 원본 데이터

- **이벤트 카탈로그 12종 전부 로깅** — photoUpload · recognitionDone · checklistEdit · matchingDone · suggestionsShown · suggestionOpened · cooked · cookedUndo · rematch · recipeBookChanged · backup · errorShown. E2E 카탈로그 테스트가 export JSON에 12종 전부 있음을 강제한다(부족·초과 시 실패).
- **수동 수정 산식 입력 완비 (ADR-0003)** — checklistEdit이 5종 kind를 전부 남긴다: uncheck(해제) · recheck(재체크) · add(추가) · substitute(치환) · **vagueDismiss(오탐복귀)** + path(typing/chip). 측정 푸터의 "수동 수정" 카운터가 이 집합을 센다.
- **원가 원장** — recognitionDone · matchingDone에 costUsd · 토큰 · model 기록. 취소된 인식 호출의 토큰도 사라지지 않는다(#36 · #42).
- **계측 격리 (ADR-0004)** — 측정 푸터는 **앱바 "냉파" 타이틀 롱프레스**로 연 그 세션에만 트리에 존재한다(#143). `?debug` URL 쿼리 트리거는 제거됐다 — 네이티브엔 URL 쿼리가 없다. 배우자 화면엔 위젯 자체가 없고(숨김이 아니라 부재), 롱프레스는 접근성 트리에도 공표되지 않는다(excludeFromSemantics).

### 배포

- **네이티브 APK 빌드·서명 성립** — `android/` 스캐폴드가 `main`에 있고, 전용 릴리스 키스토어로 서명되도록 Gradle이 배선됐으며(`key.properties` 없으면 릴리스 빌드만 시끄럽게 실패), `allowBackup=false`로 클라우드 자동 백업이 D0 초기화를 되살리지 못한다(#141). 빌드·서명 절차 정본은 [`native-apk-runbook.md`](native-apk-runbook.md).
- **데이터 연속성 = 서명에 묶인다** — 핫픽스 재설치는 **같은 릴리스 키스토어 + `versionCode` 범프**로 Android가 업데이트로 처리해야 로컬 데이터(이벤트·레시피 북)가 보존된다(#132 · #133). 디버그 키/다른 키 빌드는 신규 설치 취급으로 2주치 데이터를 전멸시킨다.
- **LLM 프록시 라이브** — 정본 URL이 재료 인식·추출·매칭 프록시를 서빙한다(#130). 네이티브 앱은 `COOKMARK_API_BASE`로 이 엔드포인트를 가리킨다. 빈 배포 404 재발 차단은 프록시에 여전히 유효(#57 · #58).
- **CI 게이트** — 매 PR · main push에서 format · analyze · test · E2E 자동 실행(#59 · #69). 안전망은 UI 관측으로 디커플(#60), go_router 트립와이어(#62).

> ⚠️ **정본 URL(프록시 엔드포인트)** — `cookmark-woosungdevs-projects.vercel.app`. `cookmark.vercel.app`은 남의 프로젝트다(Vercel 전역 네임스페이스).

## 파운더 D0 체크리스트

### D0 전 (~7/21 · 베이스라인 종료)

- [ ] **기록 초기화** — 관통 테스트가 만든 이벤트를 앱 안에서 지운다(#144). **앱바 "냉파" 타이틀 롱프레스 → 측정 푸터 → "기록 초기화" 버튼 → 확인 다이얼로그 "초기화".** 이벤트·진행 중인 재료 목록이 지워지고 **레시피 북은 보존**된다. 절차 정본은 [#41](https://github.com/woosung-dev/cookmark/issues/41). 웹의 "복사 → 사이트 데이터 삭제 → 재import" 5단계는 소멸했다.
- [ ] 초기화 후 푸터에서 **"수동 수정 0 · 인식/매칭 `-` · 누적 원가 $0.00000 · 이벤트 0"** 확인. **"이벤트 0"이 정상이다** — 웹은 재import가 backup 이벤트 1건을 남겨 "이벤트 1"이 정상이었지만, 네이티브엔 재import 자체가 없다. 초기화는 자기 구간의 이벤트를 뒤늦은 응답까지 포함해 버린다(`main_controller.dart`의 `_recordEpoch`).
- [ ] 두 기기 각각. 배우자에겐 "기록 정리"로만 설명한다(계측 존재 비공개, ADR-0004). **배우자 기기에서는 확인 후 다시 타이틀을 롱프레스해 푸터를 닫는다** — 자동 닫기는 없어(알림 하나 내렸다 올려도 발화하므로 미채택) 안 닫으면 폰을 돌려준 뒤에도 계측이 열린 채 남는다.

### D0 (7/22 · 실측 시작)

- [ ] 이 날부터 실제 요리 사용 시작 — 베이스라인 제약(기술 관통만)이 풀린다.
- [ ] 원가 · 수동 수정 모니터는 **파운더 기기에서만** 타이틀 롱프레스 제스처로.

### 파일럿 중 (D0~~8/5) — 측정 순도 규칙 (ADR-0004)

- [ ] **앱을 바꾸지 않는다** — UI · 로직 변경은 측정을 오염시킨다. 정합 리팩터(#38)는 파일럿 후. 단 catastrophic 결함은 열린 핫픽스 posture(#133)로 대응하고, 개입은 관찰 일지에 'hotfix' 유형으로 기록해 구간을 분리한다.
- [ ] 배우자는 블라인드 유지 — 계측 · 가설을 노출하지 않는다.

## 판정 지표

- **지표 1 (자발 사용)** — 주 3회 업로드 세션(photoUpload).
- **P2 킬 기준** — 사진당 수동 수정이 2주 연속 주 평균 5개를 초과하면 킬(checklistEdit 5종 합, ADR-0003). 산식은 파일럿 종료까지 불변이다.
