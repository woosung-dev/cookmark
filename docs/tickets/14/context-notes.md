## 결정: PR #24가 닫힌 원인은 DESIGN.md에 있었다

#24는 하단 탭바(`NavigationBar`)를 만들었고 이는 ADR-0001("탭 바·설정 화면 없음")과 정면 충돌한다.
그런데 **원인은 에이전트의 일탈이 아니라 DESIGN.md 자체**다 — §4가 "상단 네비바 + 하단 탭바(메인 ↔ 레시피 북)"를,
§7이 frosted 탭바 컴포넌트를 규정하고 있었다. #24는 DESIGN.md를 충실히 따랐을 뿐이다.

**판단** — 문서를 고치지 않으면 다음 에이전트가 같은 탭바를 다시 만든다. 스펙 #13의 정본 위계
("화면·UX 상세 = G1 티켓 #8 resolution")가 이 충돌을 이미 해결하고 있다:
DESIGN.md는 **디자인 언어**(색·타이포·간격·엘리베이션)의 정본이고, **화면 구조**의 정본이 아니다.
따라서 DESIGN.md §4·§7의 탭바 규정을 ADR-0001의 "헤더 링크 1개"로 정정한다. (사용자 판단 불필요 — 위계가 문서화되어 있음)

## 버그: `image.decodeImage`는 손상된 바이트에 null이 아니라 Error를 던진다

`resizeForRecognition`을 null 체크만으로 방어했는데 유닛이 잡았다 — 쓰레기 바이트를 넣으면
`decodeImage`가 내부 PSD 디코더에서 **`RangeError`**를 던진다. `RangeError`는 `Exception`이 아니라 `Error`라
`catch (e)`도 `on Exception`도 아닌 곳을 통과해 버린다. 그대로 뒀으면 손상·잘린 사진을 올렸을 때
인라인 저품질 카드가 아니라 앱이 그냥 죽었을 것이다(파일럿 D0에 배우자 기기에서 터질 종류의 버그).

→ decode를 try/catch로 감싸 `FailureReason.lowQuality`로 모은다. 테스트도 `isA<Exception>()` 대신
사유(`lowQuality`)를 못박게 조였다 — UI가 분기하는 값이 그것이므로.

## 확인: EXIF orientation은 이미 처리된다 (추측으로 고치지 않음)

실사진(`101344403muoc.jpg`, 4032×3024)은 **EXIF orientation=6**(90° 회전)을 달고 있다 — 폰 카메라 사진의 정상이다.
`img.decodeImage`가 방향을 안 굽는다면 옆으로 누운 냉장고를 Gemini에 보내게 되므로 확인했다.

**결과 — 이미 굽는다.** decodeImage가 3024×4032(세로)로 디코드하고 orientation 태그를 비운다.
`resizeForRecognition`은 576×768 정립 이미지를 낸다. 별도 `bakeOrientation` 호출 불필요 — 코드 추가 없음.
