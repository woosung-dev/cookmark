# 배우자에게 측정 지표·킬 기준을 공개하지 않는다

파일럿의 배우자(n=2의 절반)에게는 포괄 동의(사용 기록 수집·관찰 포함)만 받고, 성공 지표·킬 기준·수동 수정 계측의 존재는 공개하지 않는다(단일맹검). "수동 수정 수를 잰다"는 것을 아는 사용자는 수정을 의식해 P2 킬 기준 측정이 왜곡되고, 파일럿 중 한번 공개하면 되돌릴 수 없다. 완전 공개(투명성)와의 트레이드오프에서 측정 순도를 택했다.

## Consequences

- 배우자 온보딩은 사용법 3가지(사진→체크→오늘 할 3개 · 이거 했어요 · 일요일 백업)까지만 설명한다.
- 앱 사용 권유는 D0 온보딩 1회로 한정한다. 이후 권유가 발생하면 관찰 기록에 'prompted'로 표기해 자발 사용 지표에서 분리한다.
- 파일럿 종료 후 디브리핑에서 전체 공개한다.
- **주간 백업 export의 원본 로그 노출은 수용한다**(2026-07-16, #31 재서명). 배우자는 일요일마다 이벤트 JSON을 클립보드로 복사해 카톡으로 보내므로 `checklistEdit`·`vagueDismiss` 같은 이벤트가 눈앞을 지나간다. 그래도 단일맹검은 선다 — 포괄 동의가 이미 "사용 기록 수집"을 덮고 있어 기록의 존재 자체는 비밀이 아니고, 이 ADR이 감추는 것은 **무엇을 지표로 삼는지**(성공 지표·킬 기준·5개/장 산식)인데 원본 이벤트는 그것을 말하지 않는다. 기각한 대안 — 파운더 대행(배우자 기기를 매주 만지는 개입이라 관찰을 오염시킨다) · 파일 다운로드 전환(카톡 인앱 브라우저에서 미검증이라 백업 자체가 깨질 위험이 더 크다).

### 베이스라인 구간의 행동 제약 (2026-07-16, #31 재서명)

비공개가 아니라 **행동**을 제약하는 규칙이라 위 목록과 성격이 다르지만, 지키는 것은 같은 원칙(측정 순도)이라 여기 함께 둔다.

원설계는 "MVP 구현 기간이 자연스러운 베이스라인 구간이 된다"였다 — 앱이 없으니 베이스라인이 저절로 깨끗했다. 실제로는 순서가 뒤집혀 **앱 배포(7/16)가 베이스라인 종료(7/21)보다 앞선다.**

- **7/16~7/21은 기술 관통만 허용한다.** 폰 실사진으로 코어 루프가 도는지 확인하는 것은 폐기량을 바꾸지 않으므로 베이스라인을 깨지 않는다. **"제안대로 실제로 요리하기"는 금지** — 베이스라인 구간에 냉파를 쓰면 그 구간 폐기가 인위적으로 줄어 성공 지표 3(폐기 감소)의 비교 기준이 깎이고, 심하면 개선 방향이 뒤집혀 보인다.
- 관통으로 생긴 테스트 이벤트는 export에 섞이므로 **D0 전에 기록을 초기화**한다.
