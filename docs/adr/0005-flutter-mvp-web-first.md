# MVP를 Flutter로 구현한다 — 파일럿은 Web 빌드로 배포

승인된 디자인 문서는 Approach A의 스택으로 Next.js 단일 앱을 명시했으나, 2026-07-13 사용자 결정으로 MVP를 Flutter(Dart) 단일 코드베이스로 전환한다. 트레이드오프 — 파운더의 Flutter 학습 목표(파일럿 결과와 무관하게 회수)와 Go 이후 네이티브 앱 전환 시 코드베이스 연속성을 얻는 대신, 첫 Flutter 프로젝트의 학습곡선으로 파일럿 게이트(7/20) 이탈 위험을 감수한다. 파일럿 배포는 Web 빌드로 하여 G2 운영 규약(카톡 URL 온보딩·설치 0·즉시 수정 배포)을 그대로 유지한다.

## Considered Options

- **Next.js 유지(디자인 문서 원안)** — 검증 순도·속도 최우선. 기각 — 학습 목표를 회수하지 못하고, Flutter는 원안~디자인 문서 어디에서도 검토된 적 없는 신규 입력이었다.
- **Flutter 네이티브(설치 배포)** — 기각 — 배우자 기기 설치 마찰, 수정마다 재배포·재설치가 파일럿 요구와 충돌.
- **파일럿은 웹(Next.js), Flutter는 파일럿 기간 학습 병행** — 기각(사용자) — 냉파 코드에 Flutter를 즉시 적용하기를 원함.

## Consequences

- 게이트 규칙은 불변 — D0 = max(배포 완료, 7/20) 후 첫 저녁. 배포가 늦으면 D0가 밀리고 폐기 베이스라인·요리 일지 구간만 길어진다.
- 네이티브 Android 빌드는 파일럿 이후 같은 코드베이스의 후순위 타깃(두 기기 모두 Android — APK 직접 전달 가능).
- 웹 전제 결정들(인앱 브라우저 경고·브라우저 로컬 영속·JSON 복사 백업·debug 쿼리 파라미터 측정 푸터)은 Web 빌드이므로 전부 유효.
- LLM 프록시(인식·매칭)는 Flutter 앱과 분리된 서버리스 함수로 유지 — API 키는 클라이언트에 두지 않는다.
