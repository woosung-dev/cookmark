# 파일럿 배포 타깃을 네이티브 Android APK로 역전한다 — ADR-0005의 "파일럿은 Web 빌드로 배포" 역전

2026-07-19 사용자(파운더) 그릴링 결정으로 7/22 파일럿 배포 타깃을 **Web 빌드에서 네이티브 Android APK로 역전한다**([지도 #129](https://github.com/woosung-dev/cookmark/issues/129) · 타당성 게이트 GO [#134](https://github.com/woosung-dev/cookmark/issues/134) · 실행 스펙 [#140](https://github.com/woosung-dev/cookmark/issues/140) · 빌드 랜딩 [#141](https://github.com/woosung-dev/cookmark/issues/141)). ADR-0005의 "파일럿은 Web 빌드로 배포" 결정을 역전한다. ADR-0005 자신이 이미 "네이티브 Android는 파일럿 이후 후순위 타깃"이라 적어 **방향은 승인돼 있었고**, ADR-0009도 "Web 빌드는 파일럿 임시이며 네이티브로 전환한다"고 예고했다 — 이 ADR이 정하는 것은 **언제(파일럿 전)·어떻게(웹 폴백 없이)**다. ADR-0009가 인증 역전을, ADR-0010이 추출 역전을 정직하게 기록한 것과 같은 취지로, 이 역전과 그것이 남긴 새 제약을 여기 남긴다.

**부분 역전임에 주의** — **Flutter 단일 코드베이스 결정은 살아 있다. 역전 대상은 배포 타깃이지 스택이 아니다.** 코어 루프 코드·LLM seam 1개·단일 스토리지 모듈·화면 2개·측정 순도 경계 규칙은 전부 그대로다. 네이티브 전환은 `apps/mobile/`에 `android/` 플랫폼 폴더를 더하고(스파이크 [#134](https://github.com/woosung-dev/cookmark/issues/134) 검증 · [#141](https://github.com/woosung-dev/cookmark/issues/141) 랜딩) **같은 Dart 코드를 APK로 빌드**하는 것이다 — 앱을 새로 짜는 것이 아니다.

## 결정 계보 (2026-07-19 그릴링)

파운더가 세 갈래에서 **매번 가장 공격적인 쪽**을 택했다(3× 최공격, 재론 안 함).

- **① 파일럿 전 네이티브 역전** — ADR-0005는 "파일럿은 웹, 네이티브는 그 후"였다. 파운더는 이를 뒤집어 **파일럿 자체를 네이티브로** 시작한다. (덜 공격적 대안 = 웹으로 파일럿, 네이티브는 판정 후 — 기각.)
- **② 웹 폴백 없음** — 네이티브 파일럿이 D0(2026-07-22)에 성립하지 않으면 **웹으로 되돌아가지 않고 D0를 미룬다**. (덜 공격적 대안 = 못 맞추면 웹으로 파일럿 시작 — 기각. 폴백을 두면 네이티브가 절반만 되고 검증이 갈린다.)
- **③ 웹 종국 폐기** — 네이티브가 관통 증명되면 웹은 폐기한다(단, **증명 전 삭제 금지** — 아래 Consequences 웹 폐기 시퀀싱). (덜 공격적 대안 = 웹·네이티브 무기한 병존 — 기각.)

**동기** — 네이티브 UX·기능(카메라·갤러리 picker · 공유 시트 · 외부 브라우저 딥링크가 웹뷰보다 매끄럽다) + 파운더의 Flutter 학습·장기 로드맵(파일럿 판정과 무관하게 회수되는 자산 — ADR-0005가 Flutter를 택한 것과 같은 논리)이다. **제품 수요보다 파일럿을 네이티브로 치르는 것 자체**가 목적이다.

**게이트가 이 결정을 실행 가능하게 만들었다** — #134(네이티브 관통 스파이크)가 에뮬레이터에서 `네이티브 → 무인증 Vercel 프록시 → 실 Gemini → 재료 11개 → 제안 3개(1위 가지볶음 79%)`를 크래시 0 · 렌더 패리티(홍시 액센트 · 배지 · ADR-0007 탭바)로 통과시켜 **GO**로 닫혔다. 방향은 승인됐고 기술 스파인은 증명됐다. 에뮬레이터가 못 대신하는 실기기 전용 항목(카메라 · 실 냉장고 사진 정확도 · 카톡 파일 전송 설치)은 #65 D0 스모크로 이월됐다.

## Considered Options

- **웹 유지(ADR-0005 원안)** — 즉시 수정 배포 · 설치 0 · 카톡 URL 온보딩을 그대로 유지. 기각(파운더) — 네이티브 UX·학습 목표를 회수하지 못하고, 파일럿을 네이티브로 치르려는 동기와 충돌한다.
- **웹·네이티브 이중 배포(폴백 겸용)** — D0 안전판. 기각 — **검증 순도가 갈린다**(두 플랫폼의 UX가 섞이면 "사진 1장이 유튜브 검색보다 편한가"의 답이 오염된다). 이중 유지 비용도 n=2 파일럿 값에 안 맞는다.
- **iOS 포함** — 기각(범위 밖) — 두 기기 모두 Android라 APK 직접 전달이 성립한다. iOS는 애플 로그인 의무·서명 체계가 달라 파일럿 밖이다.

## Consequences

**잃는 것과 그 대응**

- **즉시 수정 배포 상실** — 웹은 프리빌드 재배포로 두 기기가 새로고침만 하면 됐다. 사이드로드 APK는 **수정마다 두 기기 재설치**다. 대응 = **열린 핫픽스 posture**([#133](https://github.com/woosung-dev/cookmark/issues/133)) — 완전 프리즈(사소 버그 방치)의 안티패턴 대신 앱 품질을 택하되, 개입 자체가 단일맹검 오염(ADR-0004)이므로 **'hotfix' 개입을 관찰 일지에 로그 + 분석 시 전/후 세그먼트 분리**한다(데이터를 버리지 않는다 — n=2 세션은 귀하다). 가장 있음직한 핫픽스 트리거(오형식 200 응답 시 로딩 영구 고착 — 폐기 arm #25의 실재 결함)는 **D0 전 하드닝**([#142](https://github.com/woosung-dev/cookmark/issues/142))으로 미리 없앤다.

- **계측의 웹 의존 소멸 → 네이티브 등가물로 이전**([#136](https://github.com/woosung-dev/cookmark/issues/136) · [#143](https://github.com/woosung-dev/cookmark/issues/143)) — `?debug` URL 쿼리 트리거는 네이티브에서 **구조적으로 false**다(`Uri.base`가 쿼리 없는 `file://`). 치환 = 앱바 "냉파" 타이틀 **숨은 제스처**(롱프레스 · 매 실행 기본 숨김 · 세션 한정). 기록 초기화도 브라우저 "사이트 데이터 삭제"에 의존했는데 네이티브엔 그 개념이 없다 → **앱 내 events-only 초기화 버튼**([#144](https://github.com/woosung-dev/cookmark/issues/144), 레시피 보존 · `_prefs.remove` 선택 삭제)으로 웹의 "복사 → 삭제 → 재import" 춤이 소멸한다.

- **새 제약 — 데이터 연속성이 서명에 묶인다**([#132](https://github.com/woosung-dev/cookmark/issues/132)) — 웹은 도메인만 같으면 로컬 영속이 이어졌다. 네이티브는 **같은 릴리스 키스토어로 서명해야 업데이트로 처리되어 데이터가 보존**된다(다른 키 = 신규 설치 = 2주치 로컬 데이터 전멸). 그래서 **전용 릴리스 키스토어**(디버그 키 의존 금지 — 머신이 바뀌면 서명이 바뀐다)와 `versionCode` 매 빌드 범프가 규약이 된다. 키·비밀번호 생성은 파운더 몫이다(에이전트는 비밀을 만들거나 보관하지 않는다 — 리포는 그 키를 읽어 쓰는 배선과 런북만 담는다). 동반 `android:allowBackup="false"`([#131](https://github.com/woosung-dev/cookmark/issues/131) · #136) — 클라우드 자동 백업이 D0 기록 초기화를 조용히 되살리는 구멍을 막는다.

- **온보딩이 파운더 hands-on으로 바뀐다**([#135](https://github.com/woosung-dev/cookmark/issues/135)) — 카톡 URL 공유 + 홈 화면 추가가 **APK 카톡 파일 전송 + 파운더 직접 설치**로 바뀐다. 배우자가 스스로 사이드로드하면 Play Protect "출처 불명" 경고가 마찰 + "실험용 베타" 프라이밍이 되어 맹검을 오염시킨다(웹 URL은 "평범한 웹사이트"라 이 위험이 없었다). 레시피 북 import도 파운더 대면 세션 — 배우자는 병합 UI를 보지 않는다. **온보딩 정본은 #135 resolution 코멘트**이고, #65·#41·#9 운영 문서 반영은 **[#146](https://github.com/woosung-dev/cookmark/issues/146)**의 몫이다(이 ADR은 결정만 기록한다).

**얻는 것**

- **CORS 소멸, 그러나 same-origin도 소멸** — 네이티브엔 origin이 없어 웹 CORS 프리플라이트가 사라진다. 대신 `COOKMARK_API_BASE`가 빈 문자열이면 `/api/recognize`가 죽은 상대경로가 되므로, 네이티브 빌드는 **반드시** `--dart-define=COOKMARK_API_BASE=https://cookmark-woosungdevs-projects.vercel.app`로 빌드한다(#134 핵심 함정 — Vercel 호스트는 소스에 하드코딩돼 있지 않고 컴파일 시 dart-define으로만 주입된다). 파일럿 프록시는 **무인증 Vercel 프록시**로 유지([#130](https://github.com/woosung-dev/cookmark/issues/130)) — FastAPI 컷오버(#98/#100)는 별개 트랙 · 파일럿 무의존이다.
- **export 개선 여지** — 공유 시트로 개선 가능하나 파일럿은 **클립보드 유지**(#136, D0 6일 전 세 번째 변경 회피). `share_plus`는 파일럿 후.

**웹 폐기 시퀀싱 — 증명 전 삭제 금지**

- `web/` 디렉토리 · `platform/user_agent*` 조건부 임포트 · 인앱 브라우저 경고 배너는 **아직 제거하지 않는다**. 네이티브 파일럿이 실기기에서 실제로 관통한 뒤에 정리한다(#134가 이월한 실기기 스모크가 그 관문). **증명 전 삭제는 순손해** — 네이티브가 D0에 무너지면(폴백은 없으나 D0는 밀 수 있다) 웹 코드가 디버깅·비교의 마지막 참조가 된다.
- **따라서 리포에 남은 웹 코드는 결함이나 미청소가 아니라 의도된 잔존이다.** 이 ADR이 그 근거다 — 다음 세션이 `web/`를 보고 "네이티브로 갔는데 왜 웹이 남았지"를 청소 대상으로 오해하지 않게 한다. `AGENTS.md`·`docs/coding-standards.md`가 이 문단을 가리킨다.

## 재검토·정리 트리거

- **웹 코드 제거** — 네이티브 파일럿이 실기기 관통 증명(#65 D0 스모크 통과) 후. 그 전엔 삭제 금지.
- **`share_plus` 공유 시트 export** — 파일럿 종료 후.
- **iOS · 애플 로그인** — 이 ADR 밖. iOS 네이티브 출시를 결정하면 그때 별도 ADR.
- **FastAPI 컷오버** — 파일럿과 무관한 별개 트랙(ADR-0009). 파일럿은 무인증 Vercel 프록시로 끝까지 간다.
