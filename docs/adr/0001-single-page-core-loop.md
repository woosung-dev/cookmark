# 코어 루프는 단일 페이지 — 화면 2개 구조

MVP(질문 검증기)의 코어 루프(사진 업로드→재료 체크리스트→제안)는 화면 전환 없이 한 세로 페이지의 섹션 확장/접힘으로 처리하고, 앱 전체 화면은 메인과 레시피 북 2개로 고정한다. 질문 검증기는 "사진 1장이 유튜브 검색보다 편한가"를 재는 마찰 측정 장치이므로, 앱 자체의 내비게이션 마찰이 측정에 섞이지 않도록 0에 수렴시킨다 — 남는 마찰이 곧 메커니즘의 마찰이 되게 한다.

## Considered Options

- **카드 중심(홈=보드, 시트 2종, 화면 5개)** — 재방문 경험은 최적이나, 마지막 제안이 항상 먼저 보여 냉장고 상태 변화 후 지표(선택·행동 변화)가 옛 카드에서 오염되는 문제. 기각.
- **수정 UX 주인공(손질대, 화면 6개)** — 로그 설계는 최정교하나 Effort S 초과, 그리고 수정을 즐겁게 만들수록 킬 기준(수동 수정 수)의 측정 대상 자체가 부양되는 구조적 결함. 기각.

## Consequences

- 화면 수를 줄인 대가로 단일 페이지 상태 기계(온보딩/로딩/체크리스트/제안/에러/세션 복원)의 상태 수가 늘어난다. 접힘 애니메이션·복원은 최소 구현으로 깎는다.
- 재료 재수정 시 아래 제안이 낡는 stale 문제가 생긴다. "다시 제안" 배너로 대응하고, stale 상태에서 발생한 카드 이벤트에는 stale 플래그를 달아 성공 지표 2의 순도를 로그 층위에서 방어한다.
- 레시피 북 진입점이 헤더 링크 1개라 발견성이 낮다 — n=2 파일럿에서는 감수하고, 다중 사용자(B 접근) 진입 시 첫 번째로 재검토한다.
