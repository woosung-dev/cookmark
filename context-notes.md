## 전제 재확인 (문서를 읽고 확인)

이 티켓은 **코드가 0줄**이다. 순수 문서 정합이고, 검증은 "문서가 서로·현실과 어긋나지 않는가"다. 리스크는 컴파일 에러가 아니라 **다음 세션 에이전트를 잘못 안내**하는 것이다.

**결정은 전부 이미 났다.** ADR-0011이 기록할 결정(웹→네이티브 역전·3× 최공격)은 2026-07-19 그릴링에서 파운더가 확정했고 지도 #129가 CLOSED다. 게이트 #134가 GO로 닫혀 "이제 문서에 네이티브 전제를 박아도 revert 위험이 없다"는 것이 이 티켓이 지금 실행되는 이유다(그 전엔 no-go 시 revert가 순손해라 `/to-spec`으로 미뤄뒀다). **나는 결정을 내리지 않는다 — 이미 난 결정을 리포 문서에 정직하게 기록한다.**

## 범위 경계 (가장 먼저 확정)

- **#145 (이 티켓) = 리포 트리 문서** — `docs/adr/`(0011 신규 + 0005 포인터) · `AGENTS.md` · `docs/coding-standards.md` · `CONTEXT.md`.
- **#146 = GitHub 이슈 문서** — #41 절차(이벤트 1→0 반전) · #65 D0 체크리스트 · #9 운영 규약 정정 코멘트.

## 결정 로그

### 1. ADR-0011 제목·구조는 0009·0010 역전 ADR의 관례를 그대로 따른다

리포에 역전 ADR이 이미 둘 있다 — 0009(인증 Clerk→OIDC)·0010(제목추론→URL내용). 둘 다 제목이 `# <새 결정> — <역전된 결정> 역전` 꼴이고, 첫 문단이 "부분 역전임에 주의"로 **무엇이 살아남는가**를 못박는다. ADR-0011도 같은 틀을 쓴다 — 발명하지 않는다. 살아남는 것 = **Flutter 단일 코드베이스**(역전 대상은 배포 타깃이지 스택이 아니다). 이걸 첫 문단에서 크게 못박아야 미래 독자가 "네이티브 역전 = 네이티브 앱 새로 짜기"로 오독하지 않는다.

### 2. ADR-0005 포인터는 #31 재서명 관례를 재사용한다 — 인라인 주석, 본문 삭제 없음

티켓이 명시한다 — "서술 자체는 지우지 않는다. 이 리포는 #31 재서명에서 같은 관례를 이미 썼다." 그 관례는 ADR-0005 안에 **이미 살아 있다**(14행 "**실측 (2026-07-16, #31 재서명)**" 중첩 불릿). 원 서술을 남기고 그 아래 굵은 날짜 주석으로 갱신을 단다. ADR-0011 포인터도 같은 형태 — `**역전 (2026-07-19, ADR-0011)**`. 제목은 안 바꾼다(2026-07-13 시점의 판단 기록). 영향받는 Consequences 두 줄(네이티브 후순위·웹 전제 유효)에 중첩 주석을 달고, 최상단에 배너 한 줄을 얹어 "이 ADR을 처음 여는 사람"이 스크롤 전에 역전을 알게 한다.

### 3. "웹이 왜 아직 살아 있는가"는 ADR-0011 Consequences에 둔다

AC의 "웹이 아직 살아 있고 왜"는 **웹 폐기 시퀀싱**과 같은 항목이다 — `web/`·조건부 임포트·인앱 브라우저 배너는 네이티브 파일럿 관통 후 정리(증명 전 삭제 순손해). ADR-0011 Consequences에서 이걸 명시하고, `AGENTS.md`·`coding-standards.md`에는 짧은 한 줄로 "남은 웹 코드는 결함이 아니라 의도된 잔존(ADR-0011)"을 남긴다. 두 곳에 길게 쓰면 갈라진다 — 정본은 ADR-0011, 나머지는 포인터.

### 4. CONTEXT.md 주간 백업 — 손실 모드만 바꾸고 역할은 불변

글로서리 "주간 백업"의 근거가 "인앱 브라우저 유실 보험"이다. 네이티브엔 인앱 브라우저(카톡 웹뷰)가 없어 그 손실 모드가 소멸한다. 하지만 **백업이 보험이라는 성질은 불변** — 손실 모드가 바뀔 뿐이다. 네이티브 기준 손실 모드 = 기기 유실·재설치 시 서명 불일치면 데이터 전멸(#132)·`allowBackup=false`(#131·#136)라 클라우드 자동복원 없음. 그래서 export 백업이 유일 복구 경로다. "가구 단위 합산 집계의 수집 지점" 역할은 티켓이 명시한 대로 **손대지 않는다**. `_Avoid_` 동의어 표류도 없음(글로서리 항목 자체는 그대로).

### 5. `README.md`는 AC 목록 밖이지만 고쳤다 — 티켓의 goal을 정면으로 어기는 한 줄이라

AC는 5개 파일을 열거하고 README는 없다(전역 CLAUDE.md §3 — 나열 밖은 손대지 않는다). 그런데 README 27행이 **`# Web 빌드가 우선 타깃`** 이라고 적혀 있었다 — 이 티켓이 존재하는 이유가 정확히 "새 세션 에이전트가 문서를 읽고 '우선 타깃 = Web'으로 작업을 시작하지 않게"이고(AC #5 근거 그대로), README는 그 "매 세션 읽는 파일"의 대표다. **나열 밖이라고 남기면 티켓이 자기 goal을 어긴다.**

판정 = 스코프 크립이 아니라 **goal 정합**이다. 바꾼 것은 딱 두 줄 — ① 27행 주석(우선 타깃 → 개발·E2E 편의) ② 30행 배포 절("수동 프리빌드만"이 웹 Vercel 경로를 가리켰는데 네이티브 APK가 파일럿 배포 타깃이 되어 틀린 안내가 됐다 → 런북 포인터로 교체). 본문 서술 삭제 없음. **더 넓히지 않았다** — README의 나머지(토폴로지·계약)는 이 티켓과 무관해 안 건드렸다. 사용자에게 이 판단을 표면화한다.

### 6. 검증은 "문서가 서로·현실과 안 어긋나는가"뿐 — 코드 0줄

`apps/mobile/`는 손대지 않았다. `flutter analyze`는 코드 무손상 스모크로만 돌린다(문서 변경이 Dart를 깨지 않음을 확인 — No issues found). 진짜 검증은 grep으로 남은 "우선 타깃 = Web" 서술이 전부 현재 상태로 갱신됐는지, ADR-0005↔0011 양방향 링크가 걸렸는지, 도메인 용어가 글로서리를 안 벗어났는지다. `/code-review`의 Spec 축이 AC 대조를, Standards 축이 문서 규약(§5 종결 콜론·글로서리 준수)을 본다.

## 리뷰 반영 (2축 code-review)

Spec 축 = 0 finding(AC 충실·README 포함 정당). Standards 축 = 하드 위반 0(콜론·글로서리·역전 ADR 형식 전부 통과), 판단성 지적 1 + 사소 1을 반영했다.

- **"의도된 잔존" 주장이 4곳에 중복 열거된다(Standards, Duplicated Code)** — 정본 목록·근거는 ADR-0011이고 나머지는 짧은 포인터여야 한다는 게 **내 결정 3의 목표 그대로였는데 AGENTS.md가 그걸 어겼다**(3항목 전량 재열거 + "결함 아님"까지 복제). 리포 규약이 "두 곳에 쓰면 갈라진다"(coding-standards §정본 위계)라 정확히 이 실패를 경고한다. AGENTS.md·coding-standards의 꼬리 열거를 **"정본 목록은 ADR-0011이 쥔다 / 여기서 다시 열거하면 갈라진다"** 포인터로 바꿨다. 목록이 바뀌어도 드리프트하지 않는다.
- **`allowBackup=false` 귀속이 갈렸다(Standards, 사소)** — CONTEXT.md만 `#136`, ADR-0011·context-notes는 `#131·#136`. #131이 매니페스트 하드닝(구멍 식별) 티켓이라 후자가 정확하다. CONTEXT.md를 `#131·#136`으로 맞췄다 — 문서 정합이 이 티켓의 주제라 사소해도 고친다.
- **"지도 #129 잔여" 표현이 어색하다(Spec, soft)** — #129는 CLOSED인데 "잔여"라 읽으면 열린 지도처럼 보인다. 실제로는 지도가 **fog로 남긴 미티켓 항목**(파일럿 후 정리 트랙, 웹 폐기와 커플)이다. AGENTS.md·coding-standards를 "파일럿 후 정리 트랙 — 지도 #129 잔여 fog, 미착수"로 crisp하게 바꿨다. 없는 트래킹 이슈를 지어내지 않았다(그 항목은 실제로 티켓화되지 않았다).
